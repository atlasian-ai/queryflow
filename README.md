# QueryFlow

**Visual SQL pipeline builder for accountants**

QueryFlow lets you build multi-step data transformation pipelines on a drag-and-drop canvas. Upload a spreadsheet, describe what you want in plain English, and get DuckDB SQL generated automatically. Chain nodes together, run the pipeline, and download your results — no coding required.

---

## Overview

QueryFlow is a web application that brings a node-based pipeline editor to finance and accounting workflows. Each node on the canvas represents one SQL transformation step. You write a natural language prompt describing what you want — QueryFlow uses Claude AI to generate the DuckDB SQL for you. When you run the pipeline, nodes execute in the correct order, with each node's output automatically available as an input to downstream nodes.

Uploaded CSV and XLSX files are converted to Parquet and stored in Supabase Storage. Node results are also stored as Parquet, which means large result sets are held in object storage rather than in the database.

## Architecture Overview

```
Browser
  │
  ├─ Supabase Auth (JWT)
  │
  └─ FastAPI (Railway)
       ├─ /api/auth       — user sync
       ├─ /api/sources    — CSV/XLSX upload → Parquet → Supabase Storage
       ├─ /api/pipelines  — pipeline + node + edge CRUD, SQL generation
       └─ /api/pipelines  — run trigger, run status polling, result download
            │
            ├─ PostgreSQL (Supabase)   — metadata, run status, schemas
            ├─ Redis (Railway)         — Celery broker + result backend
            └─ Celery Worker           — DuckDB execution, Parquet I/O
                   │
                   └─ Supabase Storage — Parquet files (sources + node results)
```

The FastAPI server and Celery worker run in the same Docker container on Railway, started via `start.sh`. Alembic migrations run automatically on container start.

## How It Works

### Node-Based Pipeline

Every pipeline is a directed acyclic graph (DAG) of transform nodes on a canvas. Each node has:

- **Label** — a human-readable display name
- **SQL name (slug)** — auto-generated, SQL-safe identifier used as the DuckDB view name
- **Natural language prompt** — plain English description of what this step should do
- **DuckDB SQL** — the actual query that runs (can be written manually or generated by AI)

Nodes are connected by edges. When you run a pipeline, nodes execute in topological order (Kahn's algorithm). Each node's output DataFrame is registered as a DuckDB view so downstream nodes can reference it by the node's slug.

### Natural Language to SQL

Clicking "Generate SQL" on a node sends the node's prompt to the backend. The backend identifies all upstream nodes and uploaded data sources accessible to that node, builds a schema context (table names + column types), and sends a prompt to Claude Haiku. The model returns a single DuckDB SELECT statement. You can edit the generated SQL in the inline CodeMirror editor before running.

### Execution and Storage

At run time:

1. The API creates a `pipeline_run` record and enqueues a Celery task.
2. The Celery worker downloads all uploaded data sources from Supabase Storage and registers them as DuckDB views.
3. Nodes execute in topological order. Each node's result DataFrame is:
   - Uploaded to Supabase Storage as a Snappy-compressed Parquet file at `runs/{run_id}/{node_id}.parquet`
   - Stored inline (first 200 rows) in the `node_results` table for fast preview
   - Registered as a DuckDB view for downstream nodes in the same run
4. The frontend polls the run status endpoint every 2 seconds until the run completes.

## Features

- Visual drag-and-drop pipeline canvas with dark theme and dot-grid background
- Single unified node type — every node can be a source reference or a transform
- Claude AI generates DuckDB SQL from plain English prompts
- Inline CodeMirror SQL editor with SQL syntax highlighting and oneDark theme
- Upstream node outputs automatically available as named views in downstream SQL
- Topological sort ensures correct node execution order; cycle detection raises an error
- Per-node run status badges: pending / running / success / failed
- Row count displayed on each node after a successful run
- File upload: CSV and XLSX (up to 50 MB), converted to Parquet on upload
- Data preview panel powered by AG Grid (paginated, supports large result sets)
- Download node results as CSV or XLSX
- Inline renaming of pipelines, nodes, and data sources
- Resizable node configuration panel and SQL editor
- Stop / cancel run button
- Run history (last 50 runs per pipeline)
- Supabase email/password authentication

## Getting Started

### Prerequisites

- Node.js 20+
- Python 3.12+
- A Supabase project (free tier works)
- A Redis instance (local or hosted)
- An Anthropic API key

### Backend

```bash
cd backend
python -m venv venv && source venv/bin/activate
pip install -r requirements.txt

cp .env.example .env
# Edit .env with your values (see Environment Variables below)

alembic upgrade head
uvicorn app.main:app --reload --port 8000

# In a separate terminal, start the Celery worker
celery -A app.workers.celery_app worker --loglevel=info
```

### Frontend

```bash
cd frontend
npm install

cp .env.example .env
# Edit .env with your values

npm run dev
```

### Environment Variables

**Backend (`backend/.env`)**

| Variable | Description |
|---|---|
| `SUPABASE_URL` | Your Supabase project URL |
| `SUPABASE_JWT_SECRET` | JWT secret from Supabase project settings |
| `SUPABASE_SERVICE_KEY` | Supabase service role key (for Storage access) |
| `DATABASE_URL` | PostgreSQL connection string (`postgresql+asyncpg://...`) |
| `SUPABASE_STORAGE_BUCKET` | Storage bucket name (default: `queryflow-files`) |
| `REDIS_URL` | Redis connection URL |
| `ANTHROPIC_API_KEY` | Anthropic API key for Claude |
| `FRONTEND_URL` | Frontend origin for CORS |

**Frontend (`frontend/.env`)**

| Variable | Description |
|---|---|
| `VITE_SUPABASE_URL` | Your Supabase project URL |
| `VITE_SUPABASE_ANON_KEY` | Supabase anon (public) key |
| `VITE_API_URL` | Backend API base URL |

### Supabase Setup

1. Create a new Supabase project.
2. In Storage, create a private bucket named `queryflow-files`.
3. Enable email/password auth under Authentication > Providers.
4. Use the connection pooler URL (port 6543, `?pgbouncer=true`) for `DATABASE_URL` when deploying to Railway.

### Deploying to Railway

The project ships with `railway.toml` files in both `backend/` and `frontend/`. Connect your GitHub repository to Railway, create three services (backend, frontend, Redis), and set the environment variables listed above. Railway injects `REDIS_URL` automatically for the Redis service.
